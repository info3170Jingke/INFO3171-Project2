Switch

When to use
  • - If you need to represent the switching between two states or on-off state.
  • - The difference between Switch and Checkbox is that Switch will trigger a state change directly when you toggle it, while Checkbox is generally used for state marking, which should work in conjunction with submit operation.
Two switch toggle
Three switch toggle
Basic switch toggle
Content
Toggle should always have a default value

Toggle comes with a pre-selected default state (either ON or OFF). If you need to allow users to make a choice (i.e the user needs to answer ‘yes’ or ‘no’), it’s better to use a checkbox because the checkbox doesn’t have a default value.


Write good labels

Good labels make the toggle switch easy to understand. Label makes it clear what option the toggle controls, as well as what state the toggle right now.


Any action triggered by the toggle should immediately take effect

When users interact with toggles, they shouldn’t need to click Save or Confirm button to apply a new state. For the very same reason, it’s better to avoid using toggles in long forms where other types of form fields are present, and the user needs to tap the Submit button for changes to take place. In this case, it’s safer to replace a toggle switch with a single checkbox.

Modals

The modal component provides a solid foundation for creating dialogs, popovers, lightboxes, or whatever else.
Usage
When the modal opens:
  • - focus is trapped within the modal and set to the first tabbable element.
  • - content behind a modal dialog is inert, meaning that users cannot interact with it.

Content
Body text

Modal content should be centered around task-oriented scenarios and what a user must do to complete that task. If the task is complicated or requires more than a few lines of instructional text, consider using another component or experience, such as a separate page in the workflow. If content exceeds the container height, an overflow scrollbar will appear, which may cause users to miss important context.


Capitalization and case

Modal content should be centered around task-oriented scenarios and what a user must do to complete that task. If the task is complicated or requires more than a few lines of instructional text, consider using another component or experience, such as a separate page in the workflow. If content exceeds the container height, an overflow scrollbar will appear, which may cause users to miss important context.


Modal headers

For modal headers, be brief and descriptive about the action a user must take to resolve the task. Try to use verb and noun combinations that clearly describes the dialog’s task or purpose. Header length should not exceed one line.
